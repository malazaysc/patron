# PRD Structure

Keep PRDs short, implementation-aware, and strict about scope.

## Recommended Sections

1. Title and status
2. Problem statement
3. Product goal
4. Non-goals
5. Primary user workflow
6. Success criteria
7. Functional requirements
8. UX requirements
9. System constraints
10. Task lifecycle requirements
11. QA and evidence requirements
12. Operational visibility requirements
13. Failure and recovery requirements
14. Out of scope for V1
15. Open questions
16. Acceptance checklist

## Authoring Rules

- Prefer concrete behaviors over aspirational language
- State explicit non-goals early
- Tie every requirement to observable behavior
- Define human approval points
- Define what evidence proves the system worked
- Keep architecture decisions in ADRs, not mixed into the PRD unless needed for scope clarity
